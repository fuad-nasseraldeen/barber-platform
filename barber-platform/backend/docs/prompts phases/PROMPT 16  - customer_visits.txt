Summary of what was implemented:
Summary of Changes
1. Visit creation on appointment status change
POST /appointments/:id/status – staff can set status to COMPLETED or NO_SHOW
UpdateAppointmentStatusDto – validates businessId and status
BookingService.updateAppointmentStatus() – updates appointment and creates a CustomerVisit
BookingService.cancelAppointment() – creates a CANCELLED visit when an appointment is cancelled
CustomerVisitsModule – registered in BookingModule and AppModule
2. Dashboard widgets
Returning customers – customers with 2+ completed visits in the last 30 days
Average visits per customer – total completed visits / unique customers
Customer retention rate – % of customers with 2+ visits
getVisitMetrics() in AnalyticsService – uses CustomerVisit with status COMPLETED
i18n keys added for en, ar, he
3. Customer profile and visit history
CustomersModule – GET /customers (list) and GET /customers/:id (detail)
/admin/customers – table of customers with links to profiles
/admin/customers/[id] – profile page with:
Customer info (name, email, phone, branch)
Stats (total visits, no-shows, last visit, visit frequency)
Visit history table (date, service, staff, status, price)
4. Automation rules
AutomationRule model – businessId, name, triggerType, conditions (JSON), actions (JSON)
AutomationService.evaluateVisitRules() – evaluates visit-based conditions for a customer
POST /automation/evaluate-visit-rules – API to evaluate rules
Visit-based condition types:
customer_no_show_count – e.g. { type: 'customer_no_show_count', operator: '>=', value: 2 }
last_visit_date – days since last visit (e.g. { type: 'last_visit_date', operator: '>=', value: 30 })
visit_frequency – visits per day (e.g. { type: 'visit_frequency', operator: '>=', value: 0.1 })
5. Migrations
20260309174004_add_automation_rules – adds automation_rules table
API usage examples
Update appointment status (mark completed):
POST /api/v1/appointments/:id/status{ "businessId": "...", "status": "COMPLETED" }
