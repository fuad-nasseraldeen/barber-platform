Summary of the performance optimizations:
Summary of Performance Optimizations
1. Redis caching
Business: findBySlug and findById cached (10 min TTL). Cache invalidated on update/delete.
Staff list: findAll cached (5 min TTL) when not paginated. Cache invalidated on create/update/delete.
Availability slots: Uses staff_availability_cache and Redis (1 min TTL).
Booking: Uses StaffService.findAll for cached staff list.
2. Availability engine
Uses staff_availability_cache; no per-request computation.
AvailabilityWorkerService: BullMQ worker precomputes slots.
AvailabilitySchedulerService: Daily cron at 00:05 UTC queues generation for all active staff for the booking window.
On cache miss, generation is queued and an empty result is returned.
3. Background jobs (BullMQ)
Availability: availability queue – precomputes staff slots.
Reminders: notification queue – appointment reminders.
Analytics: AnalyticsAggregationService – daily cron at 01:00.
Waitlist: processSlotAvailable on appointment cancellation.
4. Analytics tables
daily_business_stats: Appointments, revenue, new customers, waitlist per day.
daily_staff_stats: Bookings and revenue per staff per day.
daily_service_stats: Bookings and revenue per service per day.
AnalyticsAggregationService: Cron at 01:00 UTC aggregates previous day.
5. API performance
Pagination:
Staff: page, limit (default 20, max 100).
Waitlist: page, limit.
Notifications: page, limit.
Indexes:
notifications(businessId, createdAt) for list queries.
